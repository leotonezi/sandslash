# Trait Objects

## What it is

A trait object is a pointer to a value of some unknown concrete type that implements a given trait, plus a pointer to a vtable. The syntax is `dyn Trait`. Because the concrete type is erased at the call site, the compiler cannot inline or monomorphize the call — it dispatches through the vtable at runtime.

```rust
// Static dispatch — compiler knows the concrete type, can monomorphize
fn run<A: PageAuditor>(a: &A, page: &PageData, dom: &Dom) -> Vec<Finding> { ... }

// Dynamic dispatch — concrete type erased, vtable call
fn run(a: &dyn PageAuditor, page: &PageData, dom: &Dom) -> Vec<Finding> { ... }
```

## Why it exists

Static dispatch (generics) requires the compiler to know every concrete type at compile time. When the set of types is open-ended or determined at runtime, trait objects are the only option. They also reduce code size — one compiled function body instead of one copy per concrete type.

In this project auditors are registered in a `Vec` at startup and iterated at runtime. The number and order of auditors can change without recompiling callers. A generic `Vec<A: PageAuditor>` cannot hold *different* auditor types simultaneously — `Vec<Box<dyn PageAuditor>>` can.

## How it works under the hood

### Fat pointer

A `&dyn Trait` or `Box<dyn Trait>` is a *fat pointer* — two pointer-sized words:

```
[ data ptr | vtable ptr ]
  ↓           ↓
  value       +----------+
              | type_id  |  (size, alignment, drop glue, …)
              | fn audit |  → MetadataAuditor::audit
              | fn id    |  → MetadataAuditor::id
              | fn drop  |
              +----------+
```

The vtable is a static, read-only table generated by the compiler for each `impl Trait for ConcreteType`. Every virtual call goes through this indirection, so it is one pointer dereference more expensive than a direct call and cannot be inlined.

### `Box<dyn Trait>` ownership

`Box<dyn Trait>` heap-allocates the concrete value and stores the fat pointer. When the `Box` is dropped, the vtable's drop glue pointer is used to call the correct destructor for the erased type.

### Object safety

Not every trait can be used as a trait object. A trait is *object-safe* if:

1. It has no generic type parameters on its methods (the compiler cannot stamp out a vtable per monomorphization).
2. Methods do not use `Self` by value (size not known at vtable call time).
3. The trait does not require `Sized`.

`async fn` in a trait is **not** object-safe on its own — `async fn` desugars to `fn() -> impl Future<Output = …>`, and `impl Future` is a unique type per implementation. This is why `SiteAuditor` uses `#[async_trait]`, which manually boxes the future to erase its type (see `docs/rust/04-async-trait.md`).


Each concrete auditor is a zero-sized struct (e.g. `pub struct MetadataAuditor;`). `Box::new(MetadataAuditor)` allocates 0 bytes of data — the heap allocation is real, but the value is zero-sized, so only the vtable pointer matters. The `Box<dyn PageAuditor>` fat pointer is what the `Vec` stores.

### Sharing across worker tasks — `src/crawler/engine.rs:51–52`

```rust
page_auditors: Arc<Vec<Box<dyn PageAuditor>>>,
site_auditors: Arc<Vec<Box<dyn SiteAuditor>>>,
```

`Arc` wraps the entire `Vec` so all worker tasks share one allocation. For `Arc<T>` to be `Send + Sync`, `T` must also be `Send + Sync`. `Box<dyn PageAuditor>` is `Send + Sync` only when the trait bound includes `+ Send + Sync` — which `PageAuditor: Send + Sync` (mod.rs:22) provides.

### Trait definition — `src/audit/mod.rs:22–33`

```rust
pub trait PageAuditor: Send + Sync {
    fn id(&self) -> &'static str;
    fn category(&self) -> Category;
    fn audit(&self, page: &PageData, dom: &Dom) -> Vec<Finding>;
}
```

All three methods take `&self` (not `Self` by value), return concrete types, and have no generic parameters — so `PageAuditor` is object-safe. The `Send + Sync` supertraits propagate to `Box<dyn PageAuditor>`, making the fat pointer thread-safe.

## Common mistakes

**Generic method breaks object safety**

```rust
pub trait Auditor {
    fn report<W: Write>(&self, writer: W);  // NOT object-safe
}
// error: the trait `Auditor` cannot be made into an object
```

Fix: use `&mut dyn Write` instead of a generic parameter, or split into a separate non-object-safe trait.

**`Self` by value breaks object safety**

```rust
pub trait Cloneable {
    fn clone(&self) -> Self;  // NOT object-safe — Self size unknown
}
```

This is why `Clone` is not object-safe. The workaround is a `fn clone_box(&self) -> Box<dyn Cloneable>` method.

**`async fn` in trait without `#[async_trait]`**

```rust
pub trait SiteAuditor {
    async fn audit(&self) -> Vec<Finding>;  // NOT object-safe as-is
}
```

`async fn` returns `impl Future`, which is a different concrete type per implementation. Fix: `#[async_trait]` (see topic 4), which rewrites the return type to `Pin<Box<dyn Future<…> + Send>>` — a concrete, object-safe type.

**Forgetting `+ Send + Sync` on the bound**

```rust
let auditors: Vec<Box<dyn PageAuditor>> = ...;
// This compiles but Arc<Vec<Box<dyn PageAuditor>>> is !Send
// because Box<dyn PageAuditor> without + Send is !Send
```

Always write `Box<dyn Trait + Send + Sync>` (or declare `Send + Sync` as supertraits on the trait itself, as this project does).

**Performance: unnecessary trait objects**

Trait objects add a vtable indirection per call. If the full set of types is known at compile time and performance is critical, prefer an enum or generics. For auditors — a small set called once per page — the cost is negligible compared to the I/O that dominates.

## Quick reference

| Pattern | Use when |
|---|---|
| `Box<dyn Trait>` | Owned, heap-allocated, single concrete type erased |
| `Arc<dyn Trait>` | Shared ownership across threads |
| `&dyn Trait` | Borrowed, no ownership transfer |
| `Vec<Box<dyn Trait>>` | Heterogeneous collection of trait implementors |
| `Arc<Vec<Box<dyn Trait>>>` | Shared read-only auditor registry (this project) |

Object-safety checklist:

| Requirement | Why |
|---|---|
| No generic method params | Each `<T>` would need its own vtable slot |
| No `Self` by value | Size not known through fat pointer |
| No `where Self: Sized` | Opts the method out of vtable — ok for non-dyn paths |
| `async fn` needs boxing | `impl Future` is unsized from vtable perspective |
